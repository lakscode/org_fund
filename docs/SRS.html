1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) defines the functional and non-functional requirements for REstackAI, a multi-tenant real estate fund management platform. It serves as the primary reference for development, testing, and stakeholder alignment.

1.2 Scope

REstackAI enables real estate investment organizations to manage funds, properties, tenants, financial reporting (balance sheets, income statements), and team members through a web-based application. The platform supports multi-tenancy — multiple organizations operate on the same infrastructure with strict data isolation via orgId filtering.

Key capabilities:

  • Multi-organization user management with role-based access control
  • Fund lifecycle management (creation, metrics, balance sheets)
  • Property / asset tracking with tenant and unit data
  • Financial dashboard (Command Center) with KPI metrics
  • CSV-based data import pipeline for legacy system migration
  • JWT-based authentication with bcrypt password hashing

2. Overall Description

2.1 Product Perspective

REstackAI is a standalone web application composed of:

  • Frontend: React + TypeScript single-page application served via Vite (port 5173)
  • Backend: Python standard library http.server with custom regex-based routing (port 8000)
  • Database: MongoDB with PyMongo driver
  • Data Pipeline: Python CSV import script for seeding from legacy systems
Architecture Summary: Browser → Axios (JWT interceptor) → HTTP Server :8000 → Regex Router → Route Handlers → Models → MongoDB

2.2 User Classes & Roles

Users belong to one or more organizations, each with a specific role. Roles are stored in the org_roles array on the user document.

RoleKeyPermissions
Administrator admin Full access: manage members, funds, properties, settings, and organization configuration
Portfolio Manager portfolio_manager Create/edit funds and properties, view all financial data, manage fund-property linkages
Analyst analyst View funds, properties, balance sheets, and dashboards; read-only access to financial data
Member member Basic read access to assigned organization data

2.4 Constraints

  • Backend must use Python standard library http.server only — no FastAPI, Flask, Django, or ASGI frameworks
  • No ORM — direct PyMongo queries for all database operations
  • Server started with python main.py (no uvicorn or external process managers)
  • MongoDB is the sole data store — no SQL databases
  • All API responses serialized via custom MongoEncoder for ObjectId/datetime handling

3. Functional Requirements

3.1 Authentication & Authorization

IDRequirementPriority
FR-101Users shall register with email, name, and password. Passwords are hashed with bcrypt before storage.High
FR-102Users shall log in with email and password. On success, the server returns a JWT access token.High
FR-103JWT tokens shall be included in the Authorization: Bearer <token> header for all protected API routes.High
FR-104The GET /api/auth/me endpoint shall return the current user's profile including organizations and roles.High
FR-105The Axios interceptor shall automatically attach the JWT token from localStorage to all API requests.High
FR-106On receiving a 401 response (except for /auth/ endpoints), the client shall clear the token and redirect to the login page.High
FR-107The AuthContext shall store the current user, selected organization, and provide login/logout/register functions.High
FR-108Users may belong to multiple organizations. Each membership includes a role (admin, portfolio_manager, analyst, member).High

Authentication Flow

  1. User submits email + password to POST /api/auth/login
  2. Server looks up user by email, verifies password hash (Passlib bcrypt)
  3. Server generates JWT token (PyJWT) with user_id claim
  4. Client stores token in localStorage
  5. Client calls GET /api/auth/me with Bearer token
  6. Server decodes JWT, fetches user + organizations + roles
  7. Client sets AuthContext state (user, currentOrg with role)

4.6 Response Format

All API responses are JSON-encoded using MongoEncoder which handles:

  • ObjectId → string
  • datetime → ISO 8601 string

Error/empty responses follow a consistent pattern:

// Success
{ "fundId": "...", "aum": 1234.56, ... }

// No data (HTTP 200)
{ "detail": "No balance sheet data found", "noData": true }

// Validation error (HTTP 400)
{ "detail": "Missing required field: email" }

5. Data Model

MongoDB collections with their fields. All collections use ObjectId as _id. Multi-tenant collections include orgId (string) for data isolation.

5.3 Relationships

  • users ↔ organizations — Many-to-many via org_ids / org_roles arrays on user
  • organizations → funds — One-to-many via orgId
  • organizations → properties — One-to-many via orgId
  • funds ↔ properties — Many-to-many via fund_properties junction
  • properties → tenants, units, sqft — One-to-many via propertyId
  • funds → balance_sheet — One-to-many via fundId

9. Data Pipeline & Seeding

9.1 CSV Import

The seed.py script imports data from CSV files in the api/data/ directory into MongoDB collections. This supports migration from legacy systems.

9.2 User Seeding

The seed script also creates default users with pre-assigned organization memberships and roles:

SEED_USERS = [
  (email, name, password, [(org_name, role), ...]),
  ...
]

Both org_ids (legacy) and org_roles (current) arrays are populated for backward compatibility.
